# GERBIL

一个适用于工业界的真实推荐系统，涵盖从离线样本生产、模型训练到线上服务的完整链路。

GERBIL: General Efficient Recommender for Benchmarking, Inference, and Learning

通用高效推荐系统：基准评测、在线推理、离线训练

样本生产：gerbil-data

模型训练：gerbil-train

在线推理：gerbil-serving

```
组织名：gerbil-rec
   │
   ├── gerbil-data      # 离线数据处理、清洗、特征构造、样本生成
   ├── gerbil-train     # 模型训练 & 实验
   └── gerbil-serving   # 在线推理 & 部署服务
```



### 1. `gerbil-data`

**Data processing and sample generation for GERBIL recommender systems.**

### 2. `gerbil-train`

**Offline model training and evaluation for GERBIL recommender systems.**

### 3. `gerbil-serving`

**Online inference and model serving for GERBIL recommender systems.**

# 每个目录是干什么的

## 1. `configs/`
放配置，不要把参数硬编码在代码里。

建议拆成：
- `configs/data/`：数据集、路径、特征字段
- `configs/model/`：模型结构参数
- `configs/train/`：batch size、epoch、optimizer、lr
- `configs/experiment/`：某次实验的组合配置

比如：
```text
configs/
├─ data/ml-1m.yaml
├─ model/deepfm.yaml
├─ train/base.yaml
└─ experiment/deepfm_ml1m.yaml
```

---

## 2. `scripts/`
放 shell 启动脚本，方便别人直接跑。

例如：
- `train.sh`
- `eval.sh`
- `export.sh`

这样别人一看就知道怎么启动。

---

## 3. `gerbil_train/`
核心代码包，建议仓库名 `gerbil-train`，Python 包名写成 `gerbil_train`，因为 Python 目录不能用 `-`。

---

## 4. `gerbil_train/cli/`
命令行入口。

比如：
- `train.py`：启动训练
- `evaluate.py`：离线评测
- `export.py`：导出模型给 `gerbil-serving`

这是很关键的一层，方便后续做成：
```bash
python -m gerbil_train.cli.train --config ...
```

---

## 5. `gerbil_train/data/`
数据读取、样本组织、batch 构造都放这。

包括：
- dataset 定义
- dataloader
- negative sampling
- feature transform

推荐系统训练里这层通常很重要。

---

## 6. `gerbil_train/models/`
放所有模型实现。

建议先有：
- `base.py`
- `mf.py`
- `deepfm.py`
- `sasrec.py`

以后你还可以扩展：
- `din.py`
- `bert4rec.py`
- `two_tower.py`

---

## 7. `gerbil_train/losses/`
推荐系统经常会有多种 loss，单独拆出来比较清楚。

比如：
- BPR loss
- BCE loss
- sampled softmax

---

## 8. `gerbil_train/metrics/`
离线评测指标独立出来。

推荐系统常见：
- Recall@K
- NDCG@K
- HitRate@K
- AUC
- MRR

这样训练和评测逻辑更清晰。

---

## 9. `gerbil_train/trainer/`
训练控制逻辑核心。

建议包括：
- `trainer.py`
- `evaluator.py`
- `callbacks.py`
- `checkpoint.py`

把“模型定义”和“训练流程”分开，后续维护会轻松很多。

---

## 10. `gerbil_train/export/`
这个目录很重要，因为你有 **`gerbil-serving`**。

这里可以放：
- checkpoint 转换
- embedding 导出
- feature schema 导出
- serving 所需 metadata 导出

也就是训练仓和在线推理仓的衔接层。

---

## 11. `tests/`
建议一开始就有，哪怕先写少量测试。

至少测：
- dataset 输出是否正确
- metric 计算是否正确
- 模型 forward 是否能跑通
- trainer 单步训练是否正常

# configs/ 应该怎么设计 YAML

建议把 `configs/` 设计成 **“可组合、可复用、职责分离”** 的结构：  
**数据配置、模型配置、训练配置、实验配置** 分开，最后由一个 `experiment` 文件把它们组装起来。

这对 `gerbil-train` 特别合适，因为你以后一定会遇到这些组合：

- 同一份数据，试多个模型
- 同一个模型，调多个训练参数
- 同一套训练参数，跑不同数据集
- 同一模型，导出不同 serving 版本

### 二、设计原则

#### 1. 一个 YAML 只管一类事情
不要把所有内容都塞进一个大文件里。  
推荐拆成：

- `data/*.yaml`：数据来源、字段、路径、切分、特征
- `model/*.yaml`：模型结构超参数
- `train/*.yaml`：优化器、batch size、epoch、设备
- `eval/*.yaml`：评测指标、topk、评测方式
- `export/*.yaml`：导出格式、产物路径、元信息
- `experiment/*.yaml`：引用前面的配置，形成一次完整实验

---

#### 2. 实验文件只做“组装”
`experiment/*.yaml` 不要重复写太多细节，它应该像一个入口：

- 指定用哪个数据配置
- 指定用哪个模型配置
- 指定用哪个训练配置
- 指定用哪个评测配置
- 加少量本次实验特有参数

---

#### 3. 字段命名统一
比如统一使用：
- `name`
- `type`
- `path`
- `output_dir`
- `seed`

不要一会儿叫 `dataset_name`，一会儿叫 `data_name`。

### 三、推荐 YAML 设计方案

## 1. `configs/data/ml1m.yaml`
描述数据集、字段定义、路径、样本形式。

```yaml
name: ml1m
task: ranking

paths:
  train: data/ml1m/train.parquet
  valid: data/ml1m/valid.parquet
  test: data/ml1m/test.parquet
  feature_schema: data/ml1m/feature_schema.yaml

columns:
  user_id: user_id
  item_id: item_id
  label: label
  timestamp: ts

features:
  sparse:
    - user_id
    - item_id
    - gender
    - age_bucket
    - occupation
  dense:
    - item_ctr_7d
    - item_ctr_30d

label:
  type: binary
  positive_value: 1

sampling:
  negative_sampling: true
  num_negatives: 10

loader:
  batch_format: pointwise
  num_workers: 4
  pin_memory: true
```

### 说明
这里主要回答：
- 数据在哪
- 用哪些列
- 特征有哪些
- 标签是什么
- 样本组织方式是什么

---

#### 2. `configs/model/deepfm.yaml`
描述模型结构，不要写训练细节。

```yaml
name: deepfm
type: ctr

embedding_dim: 16

dnn:
  hidden_units: [256, 128, 64]
  activation: relu
  dropout: 0.1
  batch_norm: true

fm:
  enabled: true

output:
  task: binary_classification
```

### 说明
这里只写“模型长什么样”，不写 optimizer、epoch。

---

#### 3. `configs/train/base.yaml`
描述训练超参数。

```yaml
seed: 42
device: cuda

trainer:
  epochs: 10
  batch_size: 1024
  gradient_clip_norm: 5.0
  log_interval: 100
  save_interval: 1

optimizer:
  type: adam
  lr: 0.001
  weight_decay: 0.00001

scheduler:
  enabled: true
  type: cosine
  warmup_steps: 1000

loss:
  type: bce

early_stop:
  enabled: true
  monitor: auc
  mode: max
  patience: 2

checkpoint:
  dir: checkpoints/
  save_best_only: true
  monitor: auc
  mode: max

mixed_precision:
  enabled: true
```

### 说明
这里只写训练策略，不涉及具体模型。

---

#### 4. `configs/eval/ranking.yaml`
描述评测指标和评测逻辑。

```yaml
task: ranking

metrics:
  - recall@10
  - recall@50
  - ndcg@10
  - ndcg@50
  - mrr@10

evaluation:
  batch_size: 2048
  topk: [10, 50]
  group_by: user_id
  exclude_history: true
```

如果是 CTR 任务，也可以单独有 `ctr.yaml`：

```yaml
task: ctr

metrics:
  - auc
  - logloss

evaluation:
  batch_size: 4096
```

---

#### 5. `configs/export/serving.yaml`
描述导出给 `gerbil-serving` 的格式。

```yaml
format: torchscript

output:
  dir: artifacts/
  model_file: model.pt
  metadata_file: metadata.json
  feature_schema_file: feature_schema.json

serving:
  model_name: gerbil_deepfm
  model_version: v1
  signature_name: predict
```

说明

这层非常适合和 `gerbil-serving` 对接。

---

#### 6. `configs/experiment/deepfm_ml1m.yaml`
这是最关键的入口文件，负责组装。

```yaml
experiment_name: deepfm_ml1m_v1

data: configs/data/ml1m.yaml
model: configs/model/deepfm.yaml
train: configs/train/base.yaml
eval: configs/eval/ctr.yaml
export: configs/export/serving.yaml

output_dir: outputs/deepfm_ml1m_v1
seed: 42

tags:
  - baseline
  - deepfm
  - ml1m
```

说明

这个文件很适合作为命令行入口：
```bash
python -m gerbil_train.cli.train --config configs/experiment/deepfm_ml1m.yaml
```

### 四、如果你想支持“覆盖配置”
你可以在实验层覆盖部分字段，比如：

```yaml
experiment_name: deepfm_ml1m_debug

data: configs/data/ml1m.yaml
model: configs/model/deepfm.yaml
train: configs/train/debug.yaml
eval: configs/eval/ctr.yaml

overrides:
  trainer:
    epochs: 1
    batch_size: 128
  optimizer:
    lr: 0.0005

output_dir: outputs/deepfm_ml1m_debug
```

这样你可以：
- 复用原配置
- 临时改少量参数
- 不需要复制整份 YAML

# gerbil-data

**Data processing and sample generation for recommender systems.**

`gerbil-data` is the data component of **GERBIL**  
(**G**eneral **E**fficient **R**ecommender for **B**enchmarking, **I**nference, and **L**earning).

It works together with:

- `gerbil-train`: offline training and evaluation
- `gerbil-serving`: online inference and model serving

---

## Overview

`gerbil-data` is designed for building reliable and reusable data pipelines for recommender systems.

It is responsible for turning raw data into structured datasets, features, and samples that can be consumed by downstream training and serving components.

Typical workflow:

1. Ingest raw user / item / interaction data
2. Clean and normalize source tables
3. Build training and evaluation datasets
4. Generate features and samples
5. Export artifacts for `gerbil-train` and `gerbil-serving`

---

## Features

- Raw data ingestion and preprocessing
- Dataset construction for recommendation tasks
- Feature engineering and feature schema generation
- Training / validation / test split generation
- Positive / negative sample generation
- Reusable data pipeline components
- Export-ready artifacts for downstream systems

# gerbil-serving README

**Online inference and model serving for recommender systems.**

`gerbil-serving` is the online serving component of **GERBIL**  
(**G**eneral **E**fficient **R**ecommender for **B**enchmarking, **I**nference, and **L**earning).

It works together with:

- `gerbil-data`: data processing and sample generation
- `gerbil-train`: offline training and evaluation

---

## Overview

`gerbil-serving` is designed for deploying recommender models to production and serving predictions with low latency and high reliability.

It is responsible for loading exported model artifacts, preparing online features, running inference, and returning recommendation results to downstream applications.

Typical workflow:

1. `gerbil-data` prepares feature definitions and metadata
2. `gerbil-train` trains and exports model artifacts
3. `gerbil-serving` loads model artifacts and serving metadata
4. Online requests are transformed into model inputs
5. The service returns recommendation scores or ranked results

---

## Features

- Online inference for recommender models
- Low-latency model serving
- Config-driven deployment
- Model artifact loading and version management
- Online feature transformation
- Ranking / scoring API support
- Health checks and observability hooks
- Easy integration with `gerbil-data` and `gerbil-train`

## Quick Start

```bash
  --config configs/service/deepfm_local.yaml
```

```json
  "user_id": "12345",
  "candidate_item_ids": ["101", "102", "103"]
}
```

Example response:

```json

  "user_id": "12345",
  "items": [
    {"item_id": "102", "score": 0.9241},
    {"item_id": "101", "score": 0.8812},
    {"item_id": "103", "score": 0.7127}
  ],
  "model_version": "v1"
}
```
